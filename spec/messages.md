# Message Types

cmd-ipc defines seven message types. Every message carries a `type` discriminator field matching one of these exact strings.

| Type | Purpose |
|---|---|
| `register.command.request` | Announce commands a peer can execute |
| `register.command.response` | Acknowledge registration |
| `list.commands.request` | Query commands available on a peer |
| `list.commands.response` | Return the list |
| `execute.command.request` | Invoke a command on a peer |
| `execute.command.response` | Return the result (correlated via `thid`) |
| `event` | Broadcast a fire-and-forget event |

## Correlation

Request/response pairs are correlated by a `thid` (thread ID, UUIDv4 string) generated by the requester and echoed on the response.

## Privacy

Command and event IDs prefixed with `_` are **private** — they MUST NOT be included in `register.command.request` payloads sent over channels, and MUST NOT be broadcast via `event` messages to other processes. They remain local to the originating registry.

## Canonical schemas

Normative JSON Schema definitions live in `schemas/`. The TypeScript Valibot definitions at `ts/packages/cmd-ipc/src/registry/command-message-schemas.ts` and the Rust equivalents at `rust/crates/cmd-ipc/src/...` are implementations of these schemas, not substitutes for them.

When the TS and Rust implementations disagree, `schemas/` wins.
