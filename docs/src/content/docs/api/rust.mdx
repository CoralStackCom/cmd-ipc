## Prelude

```rust
use coralstack_cmd_ipc::prelude::*;
```

Brings into scope: `command`, `commands`, `ChannelError`, `Command`, `CommandChannel`, `CommandDef`, `CommandError`, `CommandRegistry`, `CommandSchema`, `Config`, `InMemoryChannel`.

## CommandRegistry

The central hub — mirrors the TypeScript `CommandRegistry` 1:1.

```rust
use coralstack_cmd_ipc::prelude::*;
use std::time::Duration;

let registry = CommandRegistry::new(Config {
    id: Some("main".into()),
    router_channel: None,
    request_ttl: Duration::from_secs(30),
    event_ttl: Duration::from_secs(5),
});
```

### Config

| Field | Type | Description |
|---|---|---|
| `id` | `Option<String>` | Registry identifier for logging |
| `router_channel` | `Option<String>` | Channel ID for escalating unknown commands |
| `request_ttl` | `Duration` | Timeout for pending requests |
| `event_ttl` | `Duration` | TTL for event deduplication |

### Methods

| Method | Mirrors TS | Notes |
|---|---|---|
| `register_command(def, handler)` | `registerCommand` | Boxes the handler internally |
| `register_channel(channel)` | `registerChannel` | Returns a driver future — spawn it on your executor |
| `execute_command::<Req, Res>(id, req)` | loose `executeCommand` | Use `::<Value, Value>` for fully dynamic dispatch |
| `execute::<C>(req)` | strict `executeCommand<K>` | `C: Command` pins request and response types |
| `emit_event(event, payload)` | `emitEvent` | Returns `Result` (serde errors surface here) |
| `add_event_listener(event, cb)` | `addEventListener` | Returns `impl FnOnce()` unsubscribe |
| `list_commands()` | `listCommands` | Returns `Vec<CommandDef>` |
| `list_channels()` | `listChannels` | Returns connected channel IDs |
| `dispose()` | `dispose` | Closes all channels, drops commands |
| `id()` | `id` property | Registry identifier |

<Aside type="note">
  Commands owned by a channel are cleaned up automatically when the channel closes. There is no `unregister_command`, matching the TS reference. For dynamic lifecycle-scoped command groups, implement a custom `CommandChannel` that advertises its commands on connect.
</Aside>

## Command trait

Strict-mode typing — the Rust equivalent of `CommandSchemaMap`.

```rust
use coralstack_cmd_ipc::prelude::*;
use schemars::JsonSchema;
use serde::{Deserialize, Serialize};

#[derive(Deserialize, Serialize, JsonSchema)]
struct AddReq { a: i64, b: i64 }

struct Add;

impl Command for Add {
    const ID: &'static str = "math.add";
    type Request = AddReq;
    type Response = i64;
}
```

Then execute in strict mode:

```rust
let result: i64 = registry.execute::<Add>(AddReq { a: 2, b: 3 }).await?;
```

The `#[command]` / `#[commands]` macros generate the `Command` impl automatically — see [Defining Commands](/cmd-ipc/getting-started/rust/defining-commands/).

## CommandChannel trait

```rust
#[async_trait]
pub trait CommandChannel: Send + Sync {
    fn id(&self) -> &str;
    async fn send(&self, message: Message) -> Result<(), ChannelError>;
    fn subscribe(&self) -> Receiver<Message>;
    async fn close(&self) -> Result<(), ChannelError>;
}
```

Built-in implementation:

- **`InMemoryChannel`** — `InMemoryChannel::pair(local_id, peer_id)` returns two `Arc<InMemoryChannel>` halves wired together. Ideal for same-process tests and single-process multi-registry setups. See [InMemoryChannel](/cmd-ipc/getting-started/rust/channels/in-memory-channel/).

Implement the trait for custom transports (WebSocket, gRPC, stdio, etc.). The wire `Message` type is byte-identical to the TypeScript union — see [Protocol](/cmd-ipc/introduction/protocol/).

## Macros

Re-exported from `coralstack-cmd-ipc-macros`. See [Defining Commands](/cmd-ipc/getting-started/rust/defining-commands/) for full usage.

- `#[command("id", description = "...")]` — attribute on a free `async fn` or an `impl` method
- `#[commands]` — attribute on an `impl` block; registers every `#[command]`-tagged method via `MyService.register_all(&registry).await?`

## Errors

| Type | Purpose |
|---|---|
| `CommandError` | Returned from command handlers; serialized as `ExecuteError` on the wire |
| `ChannelError` | Transport-level failures |
| `ExecuteErrorCode` | Enum of wire-level error codes (`CommandNotFound`, `ValidationError`, etc.) |
| `RegisterErrorCode` | Enum of registration error codes (`CommandAlreadyRegistered`) |

## MCP adapter

`coralstack-cmd-ipc-mcp::McpServerChannel` implements `CommandChannel`, so it registers on the registry like any other channel. It wraps an `rmcp` server handler internally.

```rust
use std::sync::Arc;
use coralstack_cmd_ipc_mcp::McpServerChannel;

let mcp = Arc::new(McpServerChannel::new("mcp"));
let driver = registry.register_channel(mcp.clone()).await?;
tokio::spawn(driver);

mcp.serve_stdio().await?;
```

Pure translation between MCP and cmd-ipc wire messages — no registry reference held. MCP `tools/list` / `tools/call` emit `ListCommandsRequest` / `ExecuteCommandRequest` through `recv()`; the registry's `ExecuteCommandResponse` comes back via `send()` and is correlated to the waiting MCP call by `thid`.

| Method | Purpose |
|---|---|
| `new(id)` | Create the channel with a given id |
| `with_implementation(name, version)` | Override the `serverInfo` reported to MCP clients |
| `with_instructions(s)` | Attach an `instructions` string for MCP clients |
| `with_timeout(Duration)` | Timeout for MCP-originated requests awaiting the registry |
| `serve(transport)` | Drive the MCP protocol over any rmcp transport (stdio, TCP, duplex, WebSocket, …) |
| `serve_stdio()` | Convenience wrapper: `serve(rmcp::transport::io::stdio())` |
| `into_handler()` | Returns an rmcp `ServerHandler` — use this to plug into HTTP frameworks (axum, actix, warp, …) as a per-session handler factory |

Stdio ships in the crate by default. For HTTP or any other transport, enable the relevant rmcp feature in your own `Cargo.toml` and pass the transport to `serve()` (or use `into_handler()` for HTTP's per-session model).

Private commands (prefixed with `_`) are never exposed as tools.
