This guide walks you through building a simple application with commands running across a process boundary. Pick your language below — the rest of the page follows your choice.

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    A calculator where math operations run in a **Web Worker** off the main thread. The main thread calls worker commands with full type safety, events broadcast results, and private commands stay local.

  <TabItem label="Rust" icon="seti:rust">

The root registry calls worker commands through the channel, events broadcast across the pair, and private commands stay local. ## Step 1: Define Command Schemas

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    First, define the schemas for your commands. These provide both TypeScript types and runtime validation. Use `defineIds` to create type-safe command ID constants:

    ```typescript
    // src/ipc/command-schema.ts
    import * as v from 'valibot'
    import type { CommandSchemaMap, InferCommandSchemaMapType } from '@coralstack/cmd-ipc'
    import { defineIds } from '@coralstack/cmd-ipc'

    export const CalcCommandSchema = {
      'calc.add': {
        description: 'Adds two numbers together',
        request: v.object({ a: v.number(), b: v.number() }),
        response: v.object({ result: v.number() }),
      },
      'calc.multiply': {
        description: 'Multiplies two numbers together',
        request: v.object({ a: v.number(), b: v.number() }),
        response: v.object({ result: v.number() }),
      },
      // Private command — underscore prefix means it won't be shared with other processes
      '_main.log': {
        description: 'Internal logging command (private to main process)',
        request: v.object({
          level: v.picklist(['debug', 'info', 'warn', 'error']),
          message: v.string(),
        }),
      },
    } as const satisfies CommandSchemaMap

    export const CommandIDs = defineIds(CalcCommandSchema)

    type Commands = InferCommandSchemaMapType<typeof CalcCommandSchema>
    export type CommandRequest<K extends keyof Commands> = Commands[K]['request']
    export type CommandResponse<K extends keyof Commands> = Promise<Commands[K]['response']>
    ```

    <Aside type="tip">
      Use `as const satisfies CommandSchemaMap` to get the best of both worlds: literal types for inference and type checking against the schema interface.
    </Aside>
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    In Rust, the **request/response types are the schema**. `schemars` derives a JSON Schema equivalent to the Valibot one, and the `#[command]` macro wires it into the registry.

    ```rust
    // src/ipc.rs
    use schemars::JsonSchema;
    use serde::{Deserialize, Serialize};

    #[derive(Deserialize, Serialize, JsonSchema)]
    pub struct CalcReq { pub a: f64, pub b: f64 }

    #[derive(Deserialize, Serialize, JsonSchema)]
    pub struct CalcRes { pub result: f64 }

    #[derive(Deserialize, Serialize, JsonSchema)]
    pub struct LogReq {
        pub level: String,  // "debug" | "info" | "warn" | "error"
        pub message: String,
    }
    ```

    Command IDs are plain string literals passed to `#[command]` — no separate `defineIds` step.
  </TabItem>
</Tabs>

## Step 2: Define Event Schemas

Events are fire-and-forget broadcasts to all connected processes.

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    ```typescript
    // src/ipc/event-schema.ts
    import * as v from 'valibot'
    import type { EventSchemaMap } from '@coralstack/cmd-ipc'
    import { defineIds } from '@coralstack/cmd-ipc'

    export const CalcEventSchema = {
      'worker.ready': v.object({
        workerId: v.string(),
        commandCount: v.number(),
      }),
      'computation.complete': v.object({
        commandId: v.string(),
        result: v.number(),
        durationMs: v.number(),
      }),
    } as const satisfies EventSchemaMap

    export const EventIDs = defineIds(CalcEventSchema)
    ```
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    Events in Rust are plain string IDs + JSON payloads. Define serializable payload types:

    ```rust
    // src/events.rs
    use serde::{Deserialize, Serialize};

    #[derive(Deserialize, Serialize)]
    pub struct WorkerReady {
        pub worker_id: String,
        pub command_count: u32,
    }

    #[derive(Deserialize, Serialize)]
    pub struct ComputationComplete {
        pub command_id: String,
        pub result: f64,
        pub duration_ms: u64,
    }
    ```

    Emit with `registry.emit_event("worker.ready", WorkerReady { ... })?`.
  </TabItem>
</Tabs>

## Step 3: Create Command Handlers

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    Use the `@Command` decorator. The decorator only takes the command ID — descriptions live in the schema:

    ```typescript
    // src/services/calc-service.ts
    import { Command } from '@coralstack/cmd-ipc'
    import type { CommandRequest, CommandResponse } from '../ipc/command-schema'
    import { CommandIDs } from '../ipc/command-schema'

    export class CalcService {
      @Command(CommandIDs.CALC_ADD)
      async add({ a, b }: CommandRequest<typeof CommandIDs.CALC_ADD>): CommandResponse<typeof CommandIDs.CALC_ADD> {
        return { result: a + b }
      }

      @Command(CommandIDs.CALC_MULTIPLY)
      async multiply({ a, b }: CommandRequest<typeof CommandIDs.CALC_MULTIPLY>): CommandResponse<typeof CommandIDs.CALC_MULTIPLY> {
        return { result: a * b }
      }
    }
    ```
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    Use the `#[commands]` attribute on an `impl` block with one `#[command]` per method:

    ```rust
    // src/services/calc_service.rs
    use coralstack_cmd_ipc::prelude::*;
    use crate::ipc::{CalcReq, CalcRes};

    pub struct CalcService;

    #[commands]
    impl CalcService {
        #[command("calc.add", description = "Adds two numbers together")]
        async fn add(&self, req: CalcReq) -> Result<CalcRes, CommandError> {
            Ok(CalcRes { result: req.a + req.b })
        }

        #[command("calc.multiply", description = "Multiplies two numbers together")]
        async fn multiply(&self, req: CalcReq) -> Result<CalcRes, CommandError> {
            Ok(CalcRes { result: req.a * req.b })
        }
    }
    ```
  </TabItem>
</Tabs>

## Step 4: Set Up the Worker

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    Create a Web Worker that registers the command handlers. The worker waits for the main thread to send a MessagePort via `postMessage`:

    ```typescript
    // src/workers/calc.worker.ts
    import 'reflect-metadata'
    import { CommandRegistry, MessagePortChannel, registerCommands } from '@coralstack/cmd-ipc'
    import { CalcCommandSchema, CommandIDs } from '../ipc/command-schema'
    import { CalcEventSchema, EventIDs } from '../ipc/event-schema'
    import { CalcService } from '../services/calc-service'

    const registry = new CommandRegistry({
      id: 'calc-worker',
      schemas: { commands: CalcCommandSchema, events: CalcEventSchema },
    })

    registerCommands([new CalcService()], registry)

    self.onmessage = (event: MessageEvent) => {
      if (event.data.type === 'init' && event.ports[0]) {
        const port = event.ports[0]
        const channel = new MessagePortChannel('main', port)
        registry.registerChannel(channel)
        channel.start()

        registry.emitEvent(EventIDs.WORKER_READY, {
          workerId: 'calc-worker',
          commandCount: 2,
        })
      }
    }
    ```
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    In Rust, the "worker" is another `CommandRegistry` in the same (or a separate) process. Wire them with `InMemoryChannel::pair` and spawn the driver futures on your executor. This keeps the example minimal — the same pattern works across real process boundaries once you implement a transport.

    ```rust
    // src/worker.rs
    use coralstack_cmd_ipc::prelude::*;
    use crate::services::calc_service::CalcService;

    pub async fn spawn_worker(channel: Arc<dyn CommandChannel>) -> anyhow::Result<CommandRegistry> {
        let worker = CommandRegistry::new(Config {
            id: Some("calc-worker".into()),
            router_channel: Some("main".into()),
            ..Default::default()
        });

        let driver = worker.register_channel(channel).await?;
        tokio::spawn(driver);

        CalcService.register_all(&worker).await?;

        worker.emit_event("worker.ready", serde_json::json!({
            "worker_id": "calc-worker",
            "command_count": 2,
        }))?;

        Ok(worker)
    }
    ```
  </TabItem>
</Tabs>

## Step 5: Connect from Main Thread

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    Create a `MessageChannel` and pass one port to the worker, keeping the other for the main thread's registry:

    ```typescript
    // src/main.ts
    import 'reflect-metadata'
    import { CommandRegistry, MessagePortChannel } from '@coralstack/cmd-ipc'
    import { CalcCommandSchema, CommandIDs } from './ipc/command-schema'
    import { CalcEventSchema, EventIDs } from './ipc/event-schema'

    async function main() {
      const worker = new Worker(
        new URL('./workers/calc.worker.ts', import.meta.url),
        { type: 'module' }
      )

      const channel = new MessageChannel()

      const registry = new CommandRegistry({
        id: 'main',
        schemas: { commands: CalcCommandSchema, events: CalcEventSchema },
      })

      registry.registerCommand(CommandIDs._MAIN_LOG, async ({ level, message }) => {
        console[level](`[Main] ${message}`)
      })

      registry.onEvent(EventIDs.WORKER_READY, ({ workerId, commandCount }) => {
        console.log(`Worker ${workerId} ready with ${commandCount} commands`)
      })

      const workerChannel = new MessagePortChannel('calc-worker', channel.port1)
      await registry.registerChannel(workerChannel)

      worker.postMessage({ type: 'init' }, [channel.port2])

      await new Promise<void>((resolve) => {
        registry.onEvent(EventIDs.WORKER_READY, () => resolve())
      })

      const sum = await registry.executeCommand(CommandIDs.CALC_ADD, { a: 5, b: 3 })
      console.log('5 + 3 =', sum.result) // 8

      const product = await registry.executeCommand(CommandIDs.CALC_MULTIPLY, { a: 4, b: 7 })
      console.log('4 × 7 =', product.result) // 28

      await registry.executeCommand(CommandIDs._MAIN_LOG, {
        level: 'info',
        message: 'Calculations complete!',
      })
    }

    main()
    ```
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    Create the pair of channels, build the root registry, and call commands — they escalate through the channel to the worker:

    ```rust
    // src/main.rs
    use coralstack_cmd_ipc::prelude::*;
    use std::sync::Arc;
    use crate::ipc::{CalcReq, CalcRes};
    use crate::worker::spawn_worker;

    #[tokio::main]
    async fn main() -> anyhow::Result<()> {
        let (ch_for_root, ch_for_worker) = InMemoryChannel::pair("calc-worker", "main");

        let root = CommandRegistry::new(Config {
            id: Some("main".into()),
            ..Default::default()
        });

        // Private command — stays local to `main`
        root.register_command(
            CommandDef::new("_main.log"),
            |payload: serde_json::Value| async move {
                eprintln!("[Main] {payload}");
                Ok(serde_json::Value::Null)
            },
        ).await?;

        root.add_event_listener("worker.ready", |payload| {
            println!("Worker ready: {payload}");
        });

        let driver = root.register_channel(ch_for_root as Arc<dyn CommandChannel>).await?;
        tokio::spawn(driver);

        let _worker = spawn_worker(ch_for_worker as Arc<dyn CommandChannel>).await?;

        let sum: CalcRes = root.execute_command("calc.add", CalcReq { a: 5.0, b: 3.0 }).await?;
        println!("5 + 3 = {}", sum.result); // 8

        let product: CalcRes = root.execute_command("calc.multiply", CalcReq { a: 4.0, b: 7.0 }).await?;
        println!("4 × 7 = {}", product.result); // 28

        root.execute_command::<_, serde_json::Value>(
            "_main.log",
            serde_json::json!({ "level": "info", "message": "Calculations complete!" }),
        ).await?;

        Ok(())
    }
    ```
  </TabItem>
</Tabs>

<Aside>
  Private commands (prefixed with `_`) are never shared with connected channels. They're useful for internal operations that should stay within a single process.
</Aside>

## How It Works

<Steps>
1. **Schema Definition**: Valibot schemas (TS) or `serde` + `schemars`-derived types (Rust) define the contract for each command and event.
2. **Handler Registration**: `@Command` decorators (TS) or `#[command]` macros (Rust) bind functions to command IDs.
3. **Channel Setup**: `MessageChannel` + `MessagePortChannel` (TS) or `InMemoryChannel::pair` (Rust) provides the pipe between registries.
4. **Event Broadcasting**: Events propagate to all connected registries.
5. **Command Execution**: `executeCommand` / `execute_command` routes to the right handler automatically.
</Steps>

## Key Points

- **Schemas define the contract** — the wire format is byte-identical across TypeScript and Rust.
- **Handlers are registered** where the code should run (worker / child registry).
- **Private commands** (prefixed with `_`) stay local to the current process.
- **Events** broadcast to all connected processes without expecting a response.
