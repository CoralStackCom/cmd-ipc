`MCPChannel` enables integration with the <a href="https://modelcontextprotocol.io/" target="_blank" rel="noopener">Model Context Protocol (MCP)</a>, allowing cmd-ipc commands to be exposed as tools for AI agents. It supports both client and server modes using the MCP Streamable HTTP transport.

<Aside type="caution" title="Protocol Version">
  MCPChannel uses the **MCP protocol version 2025-11-25** by default and is designed specifically for this version. Older protocol versions (e.g., 2025-06-18) are not supported and may result in compatibility issues. Ensure your MCP clients and servers are using a compatible protocol version.
</Aside>

Use cases include:

- Exposing application commands as AI agent tools
- Connecting to remote MCP servers
- Building AI-powered applications with tool calling
- Creating MCP-compatible APIs

## Two Channel Types

MCPChannel comes in two variants:

| Channel | Use Case |
|---------|----------|
| **MCPServerChannel** | Expose cmd-ipc commands as MCP tools for AI agents to call |
| **MCPClientChannel** | Connect to remote MCP servers and use their tools as commands |

## MCPServerChannel

`MCPServerChannel` exposes your cmd-ipc commands as MCP tools. AI agents can discover and call these tools via the MCP protocol.

### Configuration

```typescript
import { MCPServerChannel } from '@coralstack/cmd-ipc'

const channel = new MCPServerChannel({
  id: 'mcp-server',
  serverInfo: {
    name: 'my-app',
    version: '1.0.0',
  },
  capabilities: {
    tools: {},
  },
  instructions: 'Optional instructions for the AI agent',
  enableSessions: true,  // Enable session management (default: true)
  timeout: 30000,        // Request timeout in ms (default: 30000)
})

await registry.registerChannel(channel)
```

### Configuration Options

| Option | Type | Required | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | `string` | Yes | - | Unique channel identifier |
| `serverInfo` | `MCPInfo` | No | `{ name: 'cmd-ipc', version: '1.0.0' }` | Server name and version |
| `capabilities` | `MCPServerCapabilities` | No | `{ tools: {} }` | Offered capabilities |
| `protocolVersion` | `string` | No | `'2025-11-25'` | MCP protocol version |
| `enableSessions` | `boolean` | No | `true` | Enable session management |
| `instructions` | `string` | No | - | Instructions sent to clients |
| `timeout` | `number` | No | `30000` | Request timeout in milliseconds |

### Handling Requests

Use `handleRequest()` in your HTTP handler to process incoming MCP requests. The method returns an `MCPHttpResponse` object containing status, headers, and body.

```typescript
    res.end()
  }
})
```

### Supported MCP Methods

| HTTP Method | MCP Method | Description |
|-------------|------------|-------------|
| POST | `initialize` | Establish session and exchange capabilities |
| POST | `notifications/initialized` | Client ready notification |
| POST | `tools/list` | List available tools (commands) |
| POST | `tools/call` | Execute a tool (command) |
| GET | - | Open SSE stream for server-initiated messages |
| DELETE | - | Terminate session |

### Session Management

When `enableSessions` is `true` (default), the server:

1. Creates a new session on `initialize` request
2. Returns session ID via `MCP-Session-Id` response header
3. Expects session ID in subsequent request headers
4. Cleans up session on DELETE request

```typescript
// Session ID is returned in the initialize response header
const initResponse = await fetch('/mcp', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    jsonrpc: '2.0',
    id: 1,
    method: 'initialize',
    params: { protocolVersion: '2025-11-25', capabilities: {}, clientInfo: { name: 'my-client', version: '1.0.0' } }
  })
})

const sessionId = initResponse.headers.get('MCP-Session-Id')

// Use session ID in subsequent requests
const toolsResponse = await fetch('/mcp', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'MCP-Session-Id': sessionId
  },
  body: JSON.stringify({ jsonrpc: '2.0', id: 2, method: 'tools/list' })
})
```

<Aside type="tip">
  Set `enableSessions: false` for stateless deployments like Cloudflare Workers where session state is not preserved between requests.
</Aside>

### SSE Support

The server supports Server-Sent Events (SSE) for responses when the client sends `Accept: text/event-stream` header. This enables streaming responses for long-running tool calls.

## MCPClientChannel

`MCPClientChannel` connects to remote MCP servers and exposes their tools as cmd-ipc commands.

### Configuration

```typescript
import { MCPClientChannel } from '@coralstack/cmd-ipc'

const channel = new MCPClientChannel({
  id: 'mcp-client',
  baseUrl: 'https://mcp.example.com',
  endpoint: '/mcp',           // Optional: MCP endpoint path (default: '/mcp')
  commandPrefix: 'remote',    // Optional: prefix for registered commands
  timeout: 30000,             // Optional: request timeout in ms (default: 30000)
  clientInfo: {
    name: 'my-app',
    version: '1.0.0',
  },
  capabilities: {},           // Optional: requested capabilities
})

await registry.registerChannel(channel)
```

### Configuration Options

| Option | Type | Required | Default | Description |
|--------|------|----------|---------|-------------|
| `id` | `string` | Yes | - | Unique channel identifier |
| `baseUrl` | `string` | Yes | - | MCP server URL (e.g., `https://mcp.example.com`) |
| `endpoint` | `string` | No | `'/mcp'` | MCP endpoint path |
| `commandPrefix` | `string` | No | - | Prefix added to registered command IDs |
| `timeout` | `number` | No | `30000` | Request timeout in milliseconds |
| `clientInfo` | `MCPInfo` | No | `{ name: 'cmd-ipc', version: '1.0.0' }` | Client name and version |
| `capabilities` | `MCPClientCapabilities` | No | `{}` | Requested capabilities |
| `protocolVersion` | `string` | No | `'2025-11-25'` | MCP protocol version |
| `openAuthBrowser` | `function` | No | - | OAuth callback to open browser for authorization |
| `tokenStorage` | `OAuthTokenStorage` | No | - | OAuth token storage for persistence |
| `oauthUrlTransformer` | `function` | No | - | Transform OAuth URLs (e.g., for CORS proxy) |

### Command Prefix

The `commandPrefix` option helps organize commands from different sources:

```typescript
// Without prefix:
// Remote tool 'calculator' -> local command 'calculator'

// With prefix 'mcp':
// Remote tool 'calculator' -> local command 'mcp.calculator'

const channel = new MCPClientChannel({
  id: 'mcp-client',
  baseUrl: 'https://mcp.example.com',
  commandPrefix: 'mcp',
})

// Execute with prefix
await registry.executeCommand('mcp.calculator', { operation: 'add', a: 5, b: 3 })
```

### Startup Flow

When `start()` is called on MCPClientChannel:

1. Sends `initialize` request to establish session
2. Captures session ID from `MCP-Session-Id` header
3. Sends `notifications/initialized` notification
4. Calls `tools/list` to discover available tools
5. Emits `register.command.request` for each tool (with optional prefix)

```typescript
const channel = new MCPClientChannel({
  id: 'mcp-client',
  baseUrl: 'https://mcp.example.com',
  commandPrefix: 'remote',
})

await registry.registerChannel(channel)

// After start(), remote tools are available as commands
const result = await registry.executeCommand('remote.some-tool', { input: 'value' })
```

### Accessing Server Information

After initialization, you can access server information:

```typescript
const channel = new MCPClientChannel({ /* config */ })
await registry.registerChannel(channel)

console.log(channel.serverInfo)        // { name: 'server-name', version: '1.0.0' }
console.log(channel.serverCapabilities) // { tools: { listChanged: true } }
console.log(channel.sessionId)          // 'uuid-session-id'
console.log(channel.protocolVersion)    // '2025-11-25'
```

## Complete Example

Here's a complete example showing both server and client channels:

### Server (Cloudflare Worker)

```typescript
import { CommandRegistry, MCPServerChannel, Command, registerCommands } from '@coralstack/cmd-ipc'

// Define commands
class MathCommands {
  @Command('math.add', 'Add two numbers')
  add({ a, b }: { a: number; b: number }): number {
    return a + b
  }

  @Command('math.multiply', 'Multiply two numbers')
  multiply({ a, b }: { a: number; b: number }): number {
    return a * b
  }
}

// Setup
let registry: CommandRegistry | null = null
let channel: MCPServerChannel | null = null

async function getRegistry() {
  if (!registry) {
    registry = new CommandRegistry({ id: 'worker' })
    registerCommands([new MathCommands()], registry)

    channel = new MCPServerChannel({
      id: 'mcp-server',
      serverInfo: { name: 'math-server', version: '1.0.0' },
      capabilities: { tools: {} },
    })

    await registry.registerChannel(channel)
  }
  return { registry, channel: channel! }
}

export default {
  async fetch(request: Request): Promise<Response> {
    const url = new URL(request.url)
    const { channel } = await getRegistry()

    if (url.pathname === '/mcp') {
      const method = request.method as 'GET' | 'POST' | 'DELETE'
      const body = method === 'POST' ? await request.json() : undefined

      const result = await channel.handleRequest(method, body, request.headers)

      return new Response(result.body, {
        status: result.status,
        headers: result.headers,
      })
    }

    return new Response('Not Found', { status: 404 })
  }
}
```

### Client (Node.js)

```typescript
import { CommandRegistry, MCPClientChannel } from '@coralstack/cmd-ipc'

const registry = new CommandRegistry({ id: 'client' })

const channel = new MCPClientChannel({
  id: 'mcp-client',
  baseUrl: 'https://math-server.example.com',
  commandPrefix: 'math',
})

await registry.registerChannel(channel)

// Now you can execute remote MCP tools as commands
const sum = await registry.executeCommand('math.math.add', { a: 5, b: 3 })
console.log(sum) // 8

const product = await registry.executeCommand('math.math.multiply', { a: 4, b: 7 })
console.log(product) // 28
```

For working with Server-Sent Events:

```typescript
import {
  createSSEStream,
  parseSSEStream,
  prefersSSE,
  SSE_CONTENT_TYPE,

// Check if client prefers SSE
if (prefersSSE(request.headers.get('Accept'))) {
  const { stream, writer } = createSSEStream()

  // Write messages to the stream
  writer.writeMessage({ jsonrpc: '2.0', id: 1, result: { data: 'value' } })

  return new Response(stream, {
    headers: { 'Content-Type': SSE_CONTENT_TYPE }
  })

}
```

## Common Issues

### Session Not Found

If you receive "Session not found" errors:

1. Ensure you're passing the `MCP-Session-Id` header from the initialize response
2. Check that sessions are enabled on the server (`enableSessions: true`)
3. For stateless environments, consider disabling sessions

### Tool Not Found

If commands aren't being exposed as tools:

1. Ensure commands are registered before the channel starts
2. Private commands (prefixed with `_`) are automatically filtered out
3. Check that the command has a valid ID and description

### Connection Timeout

If connections timeout:

1. Increase the `timeout` configuration option
2. Check network connectivity to the MCP server
3. Verify the `baseUrl` and `endpoint` are correct

## OAuth Authentication

MCPClientChannel supports OAuth 2.1 authentication for connecting to protected MCP servers. The implementation follows the <a href="https://modelcontextprotocol.io/specification/2025-11-25/basic/authorization" target="_blank" rel="noopener">MCP OAuth specification</a>.

### OAuth Configuration

```typescript
import { MCPClientChannel } from '@coralstack/cmd-ipc'

const channel = new MCPClientChannel({
  id: 'secure-mcp',
  baseUrl: 'https://mcp.example.com',

  // OAuth: Callback to open browser for authorization
  openAuthBrowser: async (authUrl: string) => {
    // Open popup or redirect user to authUrl
    // Return the authorization code from the callback
    return openOAuthPopup(authUrl)
  },

  // OAuth: Optional token storage for persistence
  tokenStorage: {
    async get(serverUrl: string) {
      return localStorage.getItem(`oauth-tokens:${serverUrl}`)
    },
    async set(serverUrl: string, tokens) {
      localStorage.setItem(`oauth-tokens:${serverUrl}`, JSON.stringify(tokens))
    },
    async clear(serverUrl: string) {
      localStorage.removeItem(`oauth-tokens:${serverUrl}`)
    }
  },

  // OAuth: Optional URL transformer (useful for CORS proxies)
  oauthUrlTransformer: (url: string) => {
    // Transform OAuth URLs if needed (e.g., through a CORS proxy)
    return url
  },
})
```

### OAuth Configuration Options

| Option | Type | Required | Description |
|--------|------|----------|-------------|
| `openAuthBrowser` | `(url: string) => Promise<string>` | No | Callback to open browser for authorization and return the auth code |
| `tokenStorage` | `OAuthTokenStorage` | No | Storage for persisting tokens across sessions |
| `oauthUrlTransformer` | `(url: string) => string` | No | Transform OAuth URLs (e.g., for CORS proxy) |

### How OAuth Works

When connecting to an MCP server that requires authentication:

1. **Initial Request**: The channel sends a request to the MCP server
2. **401 Response**: Server responds with `401 Unauthorized` and a `WWW-Authenticate` header containing the resource metadata URL
3. **Metadata Discovery**: The channel fetches:
   - Protected Resource Metadata (RFC 9728) to find authorization servers
   - Authorization Server Metadata (RFC 8414) or falls back to default endpoints
4. **Dynamic Client Registration**: If supported, registers the client dynamically (RFC 7591)
5. **Authorization**: Opens browser for user authorization using PKCE (S256)
6. **Token Exchange**: Exchanges authorization code for tokens
7. **Retry Request**: Retries the original request with the access token

### Browser OAuth Flow Example

For browser-based applications, you can use a popup window for OAuth:

```typescript
// oauth-popup.ts
export function openOAuthPopup(authUrl: string): Promise<string> {
  return new Promise((resolve, reject) => {
    // Open popup window
    const popup = window.open(
      authUrl,
      'oauth',
      'width=600,height=700,popup=1'
    )

    if (!popup) {
      reject(new Error('Popup blocked'))
      return
    }

    // Listen for OAuth callback via postMessage
    const handleMessage = (event: MessageEvent) => {
      if (event.data?.type === 'oauth-callback') {
        window.removeEventListener('message', handleMessage)
        popup.close()

        if (event.data.code) {
          resolve(event.data.code)
        } else {
          reject(new Error(event.data.error || 'OAuth failed'))
        }
      }
    }

    window.addEventListener('message', handleMessage)
  })
}

// In your callback page (e.g., /oauth/callback)
const params = new URLSearchParams(window.location.search)
const code = params.get('code')
const error = params.get('error')

if (window.opener) {
  window.opener.postMessage({
    type: 'oauth-callback',
    code,
    error,
  }, window.location.origin)
  window.close()
}
```

### Token Storage

For persistent OAuth tokens, implement the `OAuthTokenStorage` interface:

```typescript
interface OAuthTokenStorage {
  get(serverUrl: string): Promise<OAuthTokens | null>
  set(serverUrl: string, tokens: OAuthTokens): Promise<void>
  clear(serverUrl: string): Promise<void>
}

interface OAuthTokens {
  access_token: string
  token_type: string
  expires_in?: number
  refresh_token?: string
  scope?: string
}
```

### CORS Proxy for Browser

When connecting to external MCP servers from a browser, you may need a CORS proxy. Use `oauthUrlTransformer` to route OAuth requests through your proxy:

```typescript
const channel = new MCPClientChannel({
  id: 'mcp-client',
  baseUrl: '/proxy/https/mcp.example.com',  // Proxied URL

  oauthUrlTransformer: (url: string) => {
    // Route OAuth URLs through the same proxy
    const parsed = new URL(url)
    return `/proxy/${parsed.protocol.replace(':', '')}/${parsed.host}${parsed.pathname}${parsed.search}`
  },
})
```

### OAuth Handler

For advanced use cases, you can use `MCPOAuthHandler` directly:

```typescript
import { MCPOAuthHandler } from '@coralstack/cmd-ipc'

const oauthHandler = new MCPOAuthHandler(
  'https://mcp.example.com',  // Server URL
  openAuthBrowser,             // Browser callback
  tokenStorage,                // Optional token storage
  'my-app',                    // Client name for registration
  'http://localhost:3000/callback', // Redirect URI
  urlTransformer,              // Optional URL transformer
)

// Check if we have valid tokens
if (oauthHandler.hasValidTokens()) {
  const authHeader = oauthHandler.getAuthorizationHeader()
  // Use authHeader in requests
}

// Handle 401 response
const success = await oauthHandler.handleUnauthorized(response)

// Refresh tokens
const refreshed = await oauthHandler.refreshTokens()

// Clear tokens (logout)
await oauthHandler.clearTokens()
```
