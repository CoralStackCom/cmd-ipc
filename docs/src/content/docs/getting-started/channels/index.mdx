Channels are the transport layer. They abstract the underlying communication mechanism so the same command API works over Web Workers, HTTP, MCP, in-memory pipes, or anything you implement yourself.

## What is a Channel?

A channel is a bidirectional communication pipe that:

- Has a unique ID for identification
- Sends and receives messages (the protocol's seven `Message` variants)
- Signals connection close
- Can operate in client or server mode, depending on the channel type

## Language availability

| Channel | TypeScript | Rust | Use case |
|---|---|---|---|
| [MessagePortChannel](/cmd-ipc/getting-started/channels/message-port-channel/) | ✅ | — | Web Workers, Node `worker_threads`, Electron IPC |
| [HTTPChannel](/cmd-ipc/getting-started/channels/http-channel/) | ✅ | — (planned) | HTTP + NDJSON streaming |
| [InMemoryChannel](/cmd-ipc/getting-started/channels/in-memory-channel/) | — (use paired `MessageChannel`) | ✅ | Same-process registry-to-registry |
| [MCPChannel](/cmd-ipc/getting-started/channels/mcp-channel/) | ✅ | ✅ | Expose commands as MCP tools, or consume remote MCP servers |

<Aside type="note">
  A channel's wire format is byte-identical across languages. A TypeScript `HTTPChannel` server can host a Rust client once Rust's HTTP transport lands, and vice versa — the protocol is already shared.
</Aside>

## Channel Interface

Every channel implements a small interface. You can create custom transports (WebSocket, gRPC, stdio, etc.) by implementing the language-specific trait/interface.

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    ```typescript
    interface ICommandChannel {
      readonly id: string
      start(): Promise<void>
      close(): Promise<void>
      sendMessage(message: unknown): void
      on(event: 'message', listener: (message: unknown) => void): void
      on(event: 'close', listener: () => void): void
    }
    ```

    | Method | Description |
    |--------|-------------|
    | `id` | Unique identifier for this channel |
    | `start()` | Initialize the channel (called automatically by registry) |
    | `close()` | Close the channel and clean up resources |
    | `sendMessage()` | Send a message through the channel |
    | `on('message')` | Subscribe to incoming messages |
    | `on('close')` | Subscribe to channel close events |
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    ```rust
    #[async_trait]
    pub trait CommandChannel: Send + Sync {
        fn id(&self) -> &str;
        async fn send(&self, message: Message) -> Result<(), ChannelError>;
        fn subscribe(&self) -> Receiver<Message>;
        async fn close(&self) -> Result<(), ChannelError>;
    }
    ```

    | Method | Description |
    |---|---|
    | `id()` | Unique identifier for this channel |
    | `send()` | Send a typed `Message` through the channel |
    | `subscribe()` | Get a receiver for incoming messages |
    | `close()` | Close the channel and clean up resources |

    <Aside type="tip">
      Unlike TypeScript, `register_channel` returns a driver future the caller spawns. This keeps the crate runtime-agnostic — use tokio, async-std, smol, or `futures::executor::ThreadPool`.
    </Aside>
  </TabItem>
</Tabs>

<Aside type="tip">
  When implementing a custom channel, messages must follow the [Protocol](/cmd-ipc/introduction/protocol/) format. The registry handles message correlation via `id` and `thid` fields — your channel just serializes and transports the messages.
</Aside>

## Registering Channels

<Tabs syncKey="lang">
  <TabItem label="TypeScript" icon="seti:typescript">
    ```typescript
    const channel = new MessagePortChannel('worker', workerPort)
    await registry.registerChannel(channel)
    ```

    When a channel is registered:

    1. The channel's `start()` method is called
    2. For client-mode channels, remote commands are discovered
    3. The registry begins routing commands to/from the channel
  </TabItem>
  <TabItem label="Rust" icon="seti:rust">
    ```rust
    let channel: Arc<dyn CommandChannel> = /* ... */;
    let driver = registry.register_channel(channel).await?;
    tokio::spawn(driver);
    ```

    `register_channel` returns a `Future<Output = ()>` — spawn it on your executor. The future resolves when the channel closes. Commands owned by the channel are cleaned up automatically at that point.
  </TabItem>
</Tabs>
